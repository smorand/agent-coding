# agent-code MVP, Specification Document

> Generated on: 2026-05-03
> Project: agent-code
> Version: 1.0
> Status: Draft
> Type: Greenfield Specification

## 1. Executive Summary

`agent-code` is an autonomous coding agent designed to take a structured user story as input and produce a Pull Request as output. Its primary value proposition is **safety over speed**: the agent prioritizes correctness, reproducibility, and verifiability of the result over latency or model size. It is designed to run on **on-premise infrastructure** with mid-class open-weight models (typical ceiling: Qwen 3, 32B class), using a multi-model orchestration where each phase of the pipeline is served by an appropriately sized model declared in a configuration file.

The agent enforces a strict, sequential pipeline: classify, validate readiness, comprehend the codebase, plan, write end-to-end tests in isolation, implement code in a tight test-driven loop, review the result with a fresh-context reviewer agent, then open a Pull Request. The end-to-end test suite is the primary contract: no Pull Request can be opened unless 100% of E2E tests pass and `make check` is green. The agent never modifies its own tests during the implementation phase, eliminating the most common form of LLM cheating.

The MVP target is **Python projects** that follow the existing Python skill template (`Makefile` as single interface, `uv` toolchain, `pytest` test runner, `ruff` and `mypy` quality gates). Legacy projects are out of scope: every project handled by `agent-code` is assumed to be bootstrapped from the canonical Python project template. The MVP is invoked via a manual CLI (`agent-code <ticket-path-or-url>`); event-driven invocation (GitHub webhooks, etc.) is deferred.

Not applicable. This is a greenfield project.

## 3. Scope

### 3.1 In Scope

- A CLI binary `agent-code` invoked locally on a developer's machine or inside a Docker container, taking a single user story (Markdown file or ticket URL) as input.
- A pipeline of seven sequential phases (classification, Definition of Ready, comprehension, planning, E2E test writing, implementation loop, review) ending in Pull Request creation.
- A configuration file declaring, per phase, which model URL and model name to use (OpenAI-compatible API).
- Tooling for the agent: file read/edit/write, ripgrep, find, AST grep, LSP integration for Python, git, GitHub CLI, `make` invocation, free shell, plus MCP servers for documentation lookup (Context7) and web search (DuckDuckGo).
- Persistent agent state in `.agent_work/<ticket-id>/` versioned in git, used for crash recovery, audit trail, and reviewer context.
- A Definition of Ready gate: if the ticket is incomplete, the agent posts a comment listing what is missing and exits without code changes.
- Anti-cheating mechanisms: tests are written before implementation in an isolated phase; during implementation, writes to `tests/` are blocked at the tool wrapper level; the gate to open a Pull Request requires 100% of E2E tests passing, non-negotiable.
- A reviewer phase running with a fresh context, validating code quality, documentation freshness, and Pull Request template compliance.
- Mandatory documentation maintenance: `CLAUDE.md`, `.agent_docs/`, `README.md` updated as part of every Pull Request.
- Bootstrap from the canonical Python project template (the `python` skill) when a project structure is missing.

### 3.2 Out of Scope (Non-Goals)

- Legacy projects not built on the canonical Python template.
- Languages other than Python (Go, Rust, TypeScript, etc., are deferred to v2).
- Event-driven invocation (webhooks, polling) is deferred to v2; only CLI invocation is supported in MVP.
- Cloud-hosted SaaS deployment of the agent itself; the agent runs locally or inside a user-supplied Docker container.
- Use of large proprietary models (Claude Opus, Gemini 3.x, GLM 4.6); only on-premise OpenAI-compatible endpoints are supported.
- A web UI or GUI; the agent is CLI-only.
- Sub-Pull-Request decomposition (stacked PRs); one user story produces exactly one Pull Request. Decomposition is the responsibility of the human (or a separate Product Owner agent) at ticket-writing time.
- Real-time human-in-the-loop dialog; the agent communicates with humans only through written ticket comments and waits for the next event to resume.
- Maintenance of a semantic code index (RAG over the codebase); the agent navigates with grep, find, AST grep, and LSP, and relies on `CLAUDE.md` and `.agent_docs/` for high-level structure.
- Automatic escalation to a larger model when stuck; if the on-premise model fails to converge, the agent comments the ticket and stops.
- Unit tests as a mandatory deliverable; the agent decides when isolation justifies a unit test.

## 4. User Personas & Actors

| Actor | Description |
|---|---|
| **Developer** | The human who triggers the agent on a ticket, reviews the resulting Pull Request, and merges it. Responsible for ensuring the host machine or container has the necessary toolchain installed. |
| **Product Owner / Ticket Author** | The human (or upstream agent) who writes the user story Markdown using the canonical template. Not directly an actor of `agent-code`, but the quality of their input determines whether the Definition of Ready gate passes. |
| **Reviewer (human)** | The human who reviews and merges the agent's Pull Request. The agent produces output structured to make this review fast (templated PR description, traceability between tests and acceptance criteria). |
| **Operator / Platform Engineer** | The person who installs the agent, configures the model endpoints in `config.yaml`, provisions the on-premise GPU infrastructure, and maintains the Docker image. |

## 5. Usage Scenarios

### SC-001: Process a well-formed user story end to end

**Actor:** Developer
**Preconditions:**
- The current working directory is a Python project bootstrapped from the canonical template, with a clean git working tree on a non-protected branch.
- A user story Markdown file exists that conforms to the ticket template (title, description, acceptance criteria, optional infra notes).
- `config.yaml` declares valid model endpoints for all seven pipeline phases.
- All required CLI tools are installed (`make`, `git`, `gh`, `uv`, `ripgrep`, `ast-grep`, `pyright` LSP, MCP servers).

**Flow:**
1. Developer runs `agent-code path/to/ticket.md` in the project root.
2. The agent reads the ticket, runs the classification phase, identifies the project as Python.
3. The agent runs the Definition of Ready check; the ticket passes.
4. The agent reads `CLAUDE.md`, `.agent_docs/`, navigates the relevant code paths.
5. The agent writes a plan to `.agent_work/<ticket-id>/plan.md`.
6. The agent writes E2E tests under `tests/` derived from the acceptance criteria, commits them.
7. The agent enters the implementation loop: edits source files, runs `make check`, reads failures, adjusts. Writes to `tests/` are blocked.
8. The loop converges; `make check` is green and 100% of E2E tests pass.
9. The reviewer agent runs with a fresh context, returns `APPROVE`.
10. The agent updates `CLAUDE.md`, `.agent_docs/`, `README.md` as needed, commits, opens a Pull Request with the templated description.

**Postconditions:**
- A new branch `feat/<ticket-slug>` exists with all commits.
- A Pull Request is open against the default branch with the templated description.
- `.agent_work/<ticket-id>/` contains the full audit trail: plan, decisions, attempts log, final state.
- The local working tree is clean.

**Exceptions:**
- EXC-001a: One required CLI tool is missing on the host. The agent stops in the bootstrap check and prints a clear message naming the missing tool.
- EXC-001b: The model endpoint for a phase is unreachable. The agent retries with exponential backoff, then stops with a diagnostic if still failing.

### SC-002: Reject an incomplete user story

**Actor:** Product Owner (indirectly), Developer (invoker)
**Preconditions:** Same as SC-001, except the ticket is missing one or more required fields (e.g., no acceptance criteria, or vague description).

**Flow:**
1. Developer runs `agent-code path/to/incomplete-ticket.md`.
2. The agent runs classification (passes) and the Definition of Ready check.
3. The DoR phase identifies missing fields.
4. The agent writes a comment back to the ticket file (or to the linked issue if URL was provided) listing each missing or insufficient field with a one-line explanation.
5. The agent exits with code `1` (DoR failure), no code changes, no `.agent_work/` entries beyond the DoR report.

**Postconditions:**
- The ticket file or issue has a new comment from the agent.
- The repository is unchanged.

**Exceptions:**
- EXC-002a: The ticket file does not exist. The agent exits with a usage error.
- EXC-002b: The ticket URL is unreachable. The agent retries, then exits with a network error.

### SC-003: Implementation loop exhausts without success

**Actor:** Developer
**Preconditions:** Same as SC-001. The user story is well-formed but the implementation turns out to be intractable for the model (over-complex, ambiguous in practice, requires capabilities the model lacks).

**Flow:**
1. Steps 1 to 7 of SC-001 proceed normally.
2. The implementation loop fails to converge: it hits one or more of the four stop conditions (max iterations, stagnation, regression, wall-clock budget) across multiple distinct approaches.
3. The agent rolls back to the state right after the E2E tests were committed.
4. The agent writes a summary comment on the ticket: which approaches were tried, which tests resisted, the agent's hypothesis on why it is blocked, and a proposed decomposition of the user story into smaller sub-stories.
5. The agent leaves a draft Pull Request open with the E2E tests committed (so a human can pick up the implementation), labeled `agent-impl-blocked`.

**Postconditions:**
- A draft Pull Request exists with E2E tests but no implementation.
- The ticket has a comment summarizing the failed attempts and proposing decomposition.
- `.agent_work/<ticket-id>/` contains the full attempts log for forensics.

**Exceptions:**
- EXC-003a: The reviewer agent rejects the result of a successful loop. The agent retries the implementation loop once with the reviewer's feedback in context. If the second attempt still fails review, the agent treats it as exhaustion and follows SC-003 from step 3.

### SC-004: Anti-cheat enforcement during implementation

**Actor:** Developer (passive observer)
**Preconditions:** The agent is mid-implementation loop on a non-trivial user story.

**Flow:**
1. During the loop, the model emits a tool call attempting to write to a file under `tests/`.
2. The tool wrapper rejects the call and returns an error to the model: `tests/ is read-only during implementation phase; modify production code instead`.
3. The model receives the error, adjusts its strategy, modifies production code instead.
4. If the model repeatedly tries to modify tests (3 attempts within 10 iterations), the agent flags this as a stagnation signal and forces an approach reset.

**Postconditions:**
- No commit modifies any file under `tests/` between the E2E commit and the final implementation commit.
- The attempt log records every blocked attempt for the reviewer's context.

**Exceptions:**
- EXC-004a: The implementation legitimately requires adding a new test fixture (e.g., a JSON file under `tests/testdata/`). The wrapper distinguishes `tests/testdata/` (writable) from `tests/test_*.py` (read-only).

### SC-005: Crash recovery via .agent_work checkpoint

**Actor:** Developer
**Preconditions:** The agent is partway through processing a ticket and the host machine reboots, or the agent process is killed (OOM, manual interrupt, GPU loss).

**Flow:**
1. Developer reruns `agent-code path/to/ticket.md` after the host is back up.
2. The agent detects an existing `.agent_work/<ticket-id>/` directory with a non-final state.
3. The agent reads `state.json` and determines which phase it was in and what was the last successful checkpoint.
4. The agent resumes from the next step after the last checkpoint, replaying nothing that was already done.
5. Processing continues normally until completion or new failure.

**Postconditions:**
- The ticket completes (or fails predictably) as if no crash had occurred.
- The audit trail in `.agent_work/<ticket-id>/` is continuous.

**Exceptions:**
- EXC-005a: The git working tree was left dirty by the crash. The agent detects this and refuses to resume, asking the developer to either commit, stash, or discard.
- EXC-005b: The state file is corrupted. The agent reports the corruption and asks the developer whether to restart from scratch (deleting `.agent_work/<ticket-id>/`) or abort.

### SC-006: Bootstrap a project from the Python template

**Actor:** Developer
**Preconditions:** The current directory is empty or contains only a `.git/` and a single ticket file.

**Flow:**
1. Developer runs `agent-code path/to/ticket.md`.
2. The agent detects the absence of `pyproject.toml`, `Makefile`, `CLAUDE.md`.
3. The agent invokes the bootstrap routine: copies the canonical Python template, fills in project name and description from the ticket, runs `uv sync`.
4. The agent commits the template as the initial commit on a new branch.
5. The pipeline proceeds normally from the comprehension phase, now with a populated project.

**Postconditions:**
- The repository contains the canonical Python project structure.
- A first commit named "Bootstrap from agent-code Python template" exists.
- The pipeline proceeds to deliver the Pull Request as in SC-001.

**Exceptions:**
- EXC-006a: The template path declared in `config.yaml` is not found. The agent stops with a clear configuration error.

### Cross-Scenario Notes

- SC-002 and SC-001 are mutually exclusive on a given run: a ticket either passes DoR (then SC-001) or fails it (then SC-002).
- SC-003 and the reviewer rejection in SC-001 step 9 are linked via EXC-003a.
- SC-005 can interrupt any of SC-001, SC-002, SC-003, SC-006 and resume them.
- SC-004 is a sub-flow of SC-001 (and SC-003 during the loop).

## 6. Functional Requirements

### FR-001: CLI entry point

- **Description:** The agent exposes a single CLI command `agent-code` that accepts one positional argument: a path to a Markdown ticket file or a URL to a remote ticket. Supports `--config` to override the default config path, `--dry-run` to plan and write E2E tests without entering the implementation loop, and `--resume` to explicitly resume from `.agent_work/`.
- **Inputs:** Positional argument (path or URL), optional flags.
- **Outputs:** Exit code (0: PR opened, 1: DoR failed, 2: implementation exhausted, 3: configuration or system error).
- **Business Rules:** Must validate the working directory is a git repo (or is empty for bootstrap) before any other action.
- **Priority:** Must-have

### FR-002: Configuration loader

- **Description:** Reads a `config.yaml` (or `~/.config/agent-code/config.yaml`) declaring per-phase model endpoints. Each of the seven phases (classification, dor, comprehension, planning, e2e_writing, implementation, review) has its own entry with `url`, `model_name`, optional `api_key_env`, optional generation parameters (temperature, max_tokens). One additional entry `summarizer` is used for context compression in the implementation loop.
- **Inputs:** YAML configuration file.
- **Outputs:** Validated configuration object available to the orchestrator.
- **Business Rules:** All eight entries are required. The agent fails fast at startup if any is missing or unreachable. Endpoints must be OpenAI-compatible.
- **Priority:** Must-have

### FR-003: Project type classifier

- **Description:** A small-model phase that reads the project structure (presence of `pyproject.toml`, `package.json`, `Cargo.toml`, etc.) and the ticket description to confirm the project is Python and select the specialized pipeline. In MVP, only Python is supported; non-Python detection results in a clear error.
- **Inputs:** Project file listing, ticket text.
- **Outputs:** Detected project type (`python`) or error.
- **Business Rules:** Must be invoked first; deterministic on the file listing, model only used to disambiguate edge cases.
- **Priority:** Must-have

### FR-004: Definition of Ready check

- **Description:** Validates the ticket against the canonical ticket template. The template is documented in `.agent_docs/ticket-template.md`. Required fields: title (non-empty), description (at least 50 characters of meaningful content), acceptance criteria (at least one explicit testable criterion), optional infra notes (declarative requirements: database, external services).
- **Inputs:** Parsed ticket Markdown.
- **Outputs:** `READY` or `NOT_READY` with a structured list of missing or insufficient fields.
- **Business Rules:** If `NOT_READY`, the agent writes a comment on the ticket using a fixed format and exits with code 1, never proceeding to subsequent phases.
- **Priority:** Must-have

### FR-005: Codebase comprehension toolset

- **Description:** The agent has access to a fixed set of read-only tools to navigate the codebase: `read_file`, `list_dir`, `grep` (ripgrep), `find`, `ast_grep`, LSP queries (`lsp_definition`, `lsp_references`, `lsp_hover`), `git_log`, `git_blame`. It also has access to `read_url`, MCP Context7 (`query_docs`), and MCP DuckDuckGo (`search_web`). The agent always reads `CLAUDE.md` and the relevant `.agent_docs/*.md` first.
- **Inputs:** Tool calls from the model.
- **Outputs:** Structured tool results.
- **Business Rules:** No semantic index (RAG) is maintained. The agent must rely on these tools and the structured project documentation. Web search is allowed but discouraged for trivial questions answerable from local docs. **The agent does not depend on any external skill or model-side instruction file**; all coding standards, project conventions, and toolchain instructions are expected to live in the project's own `CLAUDE.md` and `.agent_docs/`, which are populated from the canonical Project Reference Template (Appendix B) at bootstrap time.
- **Priority:** Must-have

### FR-006: Planning artifact

- **Description:** The planning phase produces three Markdown files in `.agent_work/<ticket-id>/`: `plan.md` (high-level approach, files to touch, expected diff size), `todo.md` (ordered TODO list with checkboxes), `infra_needs.md` (declared requirements: services, fixtures, environment variables). The plan is committed to the branch as part of the agent's audit trail.
- **Inputs:** Ticket, comprehension findings.
- **Outputs:** Three Markdown files.
- **Business Rules:** If `infra_needs.md` declares a requirement that is not satisfiable in the current environment (e.g., a Postgres instance not present and not declared in `docker-compose.yml`), the agent stops and comments the ticket asking for the infrastructure to be provisioned.
- **Priority:** Must-have

### FR-007: E2E test writing in isolation

- **Description:** The E2E writing phase reads the ticket's acceptance criteria and the plan, then writes one or more test files under `tests/`. Tests are written in pytest style for Python projects. Each test maps to one or more acceptance criteria via comments (`# AC-1`). Tests must be runnable (can be collected by pytest) but are expected to fail at this stage. The phase commits the tests immediately with the message `Add E2E tests for <ticket-id>`.
- **Inputs:** Ticket, plan, codebase context.
- **Outputs:** New or modified test files under `tests/`, committed.
- **Business Rules:** This phase has no access to write tools for files outside `tests/`. The model invoked for this phase has a fresh context, no carryover from comprehension or planning beyond the ticket and the plan. Once committed, the test files are locked for the implementation phase.
- **Priority:** Must-have

### FR-008: Implementation loop

- **Description:** The implementation phase repeatedly: reads the current state, edits source files (writes to `tests/test_*.py` are blocked), runs `make check`, reads the output, adjusts. The phase tracks two counters: iterations within the current approach, and approaches attempted. Stop conditions: `max_iterations_per_approach` (default 30), stagnation (the same set of failing tests for `stagnation_threshold` iterations, default 5), regression (more tests failing than at the previous successful state, triggers a `git reset` to that state and a forced approach change), wall-clock budget (default 2 hours). Approach exhaustion triggers SC-003.
- **Inputs:** Plan, current code state, last `make check` output.
- **Outputs:** Commits on the feature branch, or exhaustion event.
- **Business Rules:** Writes to `tests/test_*.py` are blocked at the tool wrapper level. Writes to `tests/testdata/` and `tests/conftest.py` are allowed. The loop succeeds only when `make check` exits 0 AND `pytest` reports 100% of E2E tests passing. No partial success is accepted.
- **Priority:** Must-have

### FR-009: Multi-approach exploration

- **Description:** When the loop exhausts iterations on one approach, the agent must reset (`git reset --hard <e2e_commit>`) and try a distinctly different approach, recorded in `.agent_work/<ticket-id>/approaches/`. The minimum number of approaches before declaring exhaustion is `min_approaches` (default 3). Each approach attempt is logged with: starting hypothesis, key choices, why it failed.
- **Inputs:** Failed approach log, plan.
- **Outputs:** New approach attempt, or exhaustion (after `min_approaches` failures).
- **Business Rules:** An "approach" is meaningfully different if at least one of: chosen module structure, chosen algorithm, chosen library, chosen data model differs from prior approaches.
- **Priority:** Must-have

### FR-010: Loop context management

- **Description:** Each iteration of the loop starts with a context comprising: the ticket, the plan, the current `git diff` against the E2E commit, the last `make check` output, and the last N tool interactions (where N is configurable per the model's context size, default 20). Older interactions are compressed by the `summarizer` model into a rolling summary kept in `.agent_work/<ticket-id>/loop_summary.md`.
- **Inputs:** Iteration history.
- **Outputs:** Compressed context for the next iteration.
- **Business Rules:** Summarization runs every K iterations (default 10) on the iterations N+1 to last_summarized.
- **Priority:** Must-have

### FR-011: Reviewer agent

- **Description:** Once the implementation loop converges, a reviewer phase runs with a fresh model context. Inputs: the ticket, the plan, the full diff of the branch, the list of files modified, the test files. The reviewer produces a structured verdict: `APPROVE` or `REQUEST_CHANGES` with a list of concerns by severity (`blocking`, `suggestion`). Concerns must reference specific files and lines.
- **Inputs:** Branch diff, ticket, plan.
- **Outputs:** JSON verdict.
- **Business Rules:** Only `blocking` concerns trigger a return to the implementation phase. The implementation phase re-runs at most once with the reviewer's blocking concerns added to its context. A second review is then final: if blocking concerns remain, the agent treats it as exhaustion and follows SC-003. The reviewer always checks: (a) tests still validate the acceptance criteria, (b) `CLAUDE.md` and relevant `.agent_docs/` are updated to reflect new modules or conventions, (c) `README.md` is updated for any change in user-visible behavior, (d) PR template fields are populated.
- **Priority:** Must-have

### FR-012: Pull Request creation

- **Description:** When the reviewer approves and all gates are green, the agent: pushes the branch, creates a Pull Request via `gh pr create` with a description following the canonical PR template, and posts a link to the PR as a comment on the ticket.
- **Inputs:** Branch, ticket, plan, reviewer verdict.
- **Outputs:** Pull Request URL, ticket comment.
- **Business Rules:** PR description must follow the template literally (sections: User Story, Acceptance Criteria, Approach, E2E Tests, Notable Decisions, Out of Scope). All sections must be populated; empty sections are not allowed.
- **Priority:** Must-have

### FR-013: State persistence in .agent_work

- **Description:** Throughout all phases, the agent persists its state to `.agent_work/<ticket-id>/`. Files include: `state.json` (current phase, last checkpoint, counters), `plan.md`, `todo.md`, `infra_needs.md`, `approaches/<n>.md`, `loop_summary.md`, `attempts.log` (chronological tool calls and outcomes), `review.json`. State is updated after every completed step, not just at phase boundaries.
- **Inputs:** Internal agent events.
- **Outputs:** Files in `.agent_work/<ticket-id>/`.
- **Business Rules:** `.agent_work/` is committed to git. On resume (`--resume` or implicit detection), the agent rebuilds context from these files.
- **Priority:** Must-have

### FR-014: Bootstrap from Python template

- **Description:** When the working directory is empty (only `.git/` and the ticket), the agent runs a bootstrap step: copies the canonical Python Project Reference Template (Appendix B), fills in `pyproject.toml` from the ticket title and description, runs `uv sync`, commits as the initial commit on a new branch. The template ships with a fully populated `CLAUDE.md` and `.agent_docs/` containing all the Python coding rules, Makefile conventions, testing conventions, and documentation conventions the agent will rely on for every subsequent run.
- **Inputs:** Empty directory, ticket.
- **Outputs:** Populated project, initial commit.
- **Business Rules:** The template path is declared in `config.yaml`. Must complete cleanly before the comprehension phase begins. The template is the single source of truth for project conventions; the agent itself carries no language-specific knowledge beyond what is in the template's documentation.
- **Priority:** Must-have

### FR-015: Toolchain pre-flight check

- **Description:** Before any phase, the agent verifies that the host has all required tools installed: `make`, `git`, `gh` (authenticated), `uv`, `python` (3.11+), `ripgrep`, `ast-grep`, the configured Python LSP (`pyright` by default). MCP servers (Context7, DuckDuckGo) are checked by attempting a trivial call. If anything is missing or unreachable, the agent stops with a structured error listing each missing tool and how to install it.
- **Inputs:** Environment.
- **Outputs:** OK, or structured error.
- **Business Rules:** The error message must be actionable: state which tool is missing, the minimum version, and the installation command.
- **Priority:** Must-have

### FR-016: Documentation maintenance

- **Description:** As part of the implementation phase (or as a final pre-PR step), the agent updates `CLAUDE.md` (compact index) when new modules are introduced or conventions change, creates or updates `.agent_docs/<topic>.md` for any new significant topic, and updates `README.md` if user-visible behavior changes (new CLI commands, new API endpoints, new configuration). The reviewer phase explicitly verifies this.
- **Inputs:** Diff of the branch.
- **Outputs:** Doc file updates.
- **Business Rules:** A reviewer flagging missing doc updates is a `blocking` concern.
- **Priority:** Must-have

### FR-017: Audit trail commit

- **Description:** The agent commits `.agent_work/<ticket-id>/` to the feature branch, separate from code commits, so the audit trail travels with the PR. The commit message is `Audit trail for <ticket-id>`.
- **Inputs:** State files.
- **Outputs:** Commit on feature branch.
- **Business Rules:** Performed once at the end, before PR creation. Sensitive data (API keys, tokens) must not appear in `.agent_work/`; the agent must scrub any captured environment variable values.
- **Priority:** Must-have

## 7. Non-Functional Requirements

### 7.1 Performance

- The agent is **not optimized for latency**. End-to-end processing of a small user story is expected to take 10 to 60 minutes; large user stories may take several hours. The wall-clock budget per loop is configurable (default 2 hours).
- Memory budget: the orchestrator process itself uses < 500 MB; per-phase model serving is the responsibility of the user-supplied infrastructure.

### 7.2 Security

- **Authentication**:
  - GitHub access via `gh` CLI: the host machine must be authenticated; the agent does not manage credentials.
  - Model endpoints: optional bearer token per endpoint, sourced from environment variable named in `config.yaml` (e.g., `api_key_env: VLLM_TOKEN`). Never logged, never written to `.agent_work/`.
- **Sandboxing**: The agent runs with full shell access. The recommended deployment is inside a Docker container with the repository volume-mounted; a Docker image is provided. Direct host execution is supported but the developer assumes responsibility for the blast radius.
- **Secret protection**: The agent must scrub any environment variable values from `.agent_work/` artifacts before committing the audit trail. A configurable denylist of variable names (default: `*_KEY`, `*_TOKEN`, `*_SECRET`, `*_PASSWORD`) is enforced.
- **Tests are read-only during implementation**: enforced at the tool wrapper level, prevents the agent from gaming its own success criteria.

### 7.3 Usability

- The CLI prints structured progress lines (one per phase), with sub-progress for the implementation loop (iteration count, approach count, last failing tests).
- All errors are actionable: a missing tool, an invalid config, an unreachable endpoint, an exhausted loop, all produce a clear message and a suggested next action.
- The audit trail in `.agent_work/<ticket-id>/` is human-readable Markdown and JSON, suitable for post-mortem inspection.

### 7.4 Reliability

- **Crash recovery**: at any point, a kill of the agent process must allow `agent-code --resume <ticket>` (or a plain re-invocation) to continue from the last checkpoint without redoing completed work. The state file is updated atomically (write to temp, rename).
- **Deterministic exit codes**: 0 (PR opened), 1 (DoR failed), 2 (implementation exhausted), 3 (system or configuration error).
- **No partial commits**: every commit must leave the branch in a buildable state (loop iterations may not commit; only stable checkpoints commit).

### 7.5 Observability

- **OpenTelemetry**: the agent emits OTLP traces and metrics to a configured collector. Default is a JSONL file at `.agent_work/<ticket-id>/traces.jsonl`; OTLP HTTP/gRPC is supported.
- **Trace content**: one root span per agent run, child spans per phase, grandchild spans per LLM call, per tool call, per `make` invocation. LLM spans capture: model name, endpoint, input token count, output token count, duration, cost (if computable from config). **Never the prompt or response content.**
- **Metrics**: counters for iterations, approaches, blocked tool calls, reviewer requests for changes; histograms for phase durations and LLM call latencies.
- **Logging**: structured JSON logs to stdout, mirrored to `.agent_work/<ticket-id>/agent.log`. Log levels per phase configurable.
- **Sensitive data exclusion**: prompts, responses, and any environment variable matching the secret denylist are excluded from traces and logs.

### 7.6 Deployment

- **Distribution**: a Python package installable via `uv tool install agent-code` (or `pipx install agent-code`), plus a Docker image `agent-code:latest` pre-bundling the toolchain (`make`, `git`, `gh`, `uv`, `ripgrep`, `ast-grep`, `pyright`, MCP servers).
- **Configuration**: `~/.config/agent-code/config.yaml` is the default; CLI flag `--config` overrides.
- **Required external services**: a vLLM (or compatible) instance per phase model; MCP servers for Context7 and DuckDuckGo (the agent assumes these are reachable, does not provision them).
- **No CI/CD for the target project is provisioned by the agent**: the agent only opens the PR; merging and CI are the project's own concern.

### 7.7 Scalability

- The agent processes one ticket at a time per process. Concurrent tickets require concurrent processes, each with its own working directory or git worktree.
- Model endpoints are shared across processes; rate limits are the user's infrastructure concern.
- Not designed for "agent swarms" or hundreds of concurrent jobs in MVP.

### 7.8 Maintainability

- The orchestrator is a custom Python state machine, no heavy framework dependency (no LangGraph, no CrewAI in MVP).
- Each phase is a Python class implementing a common interface: `prepare`, `run`, `checkpoint`. Adding a new phase or modifying one is local to its class.
- Tools are a flat registry of Python functions wrapped to expose an OpenAI-compatible function-calling schema.
- The codebase itself follows the canonical Python project template (eats its own dog food).

## 16. Glossary

- **Approach**: A distinct strategy attempted by the implementation loop, characterized by a specific module structure, algorithm, library, or data model. Multiple approaches may be attempted before declaring the loop exhausted.
- **DoR (Definition of Ready)**: A gate that validates whether a user story has enough information for the agent to proceed. Failure produces a ticket comment listing what is missing.
- **E2E (End-to-End) test**: An executable test that validates an acceptance criterion from the user story, in conditions as close as possible to real usage. The agent's primary success contract.
- **Iteration**: One pass of the implementation loop: edit, run `make check`, observe, decide.
- **Phase**: One of the seven sequential steps in the agent's pipeline (classification, DoR, comprehension, planning, E2E writing, implementation, review).
- **Stagnation**: A state of the loop where the same set of tests has been failing for `stagnation_threshold` consecutive iterations, indicating the current approach is not converging.
- **Regression**: A state where more tests fail in the current iteration than in a previous successful state, triggering a `git reset` to that state.
- **Reviewer**: A separate agent run (fresh context, possibly same model) that verifies the implementation result against the ticket and project standards. Returns `APPROVE` or `REQUEST_CHANGES`.
- **Anti-cheat**: The set of mechanisms preventing the agent from gaming its own success criteria. Primarily: locking `tests/test_*.py` after the E2E writing phase, and verifying the E2E commit SHA before opening the PR.
- **Project Reference Template**: The canonical empty Python project (Appendix B) shipped with `agent-code`. Used by the bootstrap phase (FR-014) and read by every subsequent comprehension phase. Single source of truth for project conventions.
- **Ticket Template**: The canonical user story Markdown structure (Appendix A) that the Definition of Ready check (FR-004) validates against.
- **`agent-impl-blocked` (GitHub label)**: A repository label automatically applied to draft Pull Requests opened in the SC-003 outcome (loop exhaustion). Indicates that `agent-code` produced the E2E tests but could not converge on a passing implementation across all attempted approaches, and that human intervention or ticket re-decomposition is required. The label is created on the repository on first use if it does not already exist (auto-provisioned by FR-012).

## Appendix A: Ticket Template

This appendix defines the canonical Markdown structure that every ticket consumed by `agent-code` must follow. The Definition of Ready phase (FR-004) validates incoming tickets against this template and rejects any ticket missing a required section or providing an empty section.

> **Implementation status**: this template is materialized in `vars/ticket-template/` of this repository, with a blank version (`ticket-blank.md`), a READY example (`ticket-example-ready.md`), and a NOT_READY example (`ticket-example-not-ready.md`) usable as test fixtures.

### A.1 Filename and location

- A ticket is a Markdown file with extension `.md`.
- When a URL is provided to the CLI, the agent fetches the linked Markdown body (typically a GitHub Issue body) and treats it as if it were a local file.
- Recommended local convention: tickets live under `tickets/` at the project root, named `<slug>.md` where the slug becomes the ticket id.

### A.2 Required structure

A valid ticket has the following layout. Required sections are marked with `(REQUIRED)`; optional sections may be omitted entirely (omitting their header).

```markdown
---
id: <ticket-slug>
title: <one-line title, 5 to 80 chars>
created: <ISO-8601 date>
author: <name or handle>
labels: [<optional, comma-separated labels>]
---

# <ticket-title>

## Description (REQUIRED)

<At least 50 characters of meaningful prose describing the change being requested,
its motivation, and the user-visible outcome. Do not paste code or schemas here;
those go in dedicated sections below.>

## Acceptance Criteria (REQUIRED)

<At least one criterion. Each criterion is one bullet that begins with a stable
identifier (AC-1, AC-2, ...). Each criterion must be testable: it should be
possible to write at least one E2E test that observes a concrete outcome.>

- AC-1: <testable criterion>
- AC-2: <testable criterion>
- AC-3: <testable criterion>

## Out of Scope

<Optional. Bullet list of things deliberately excluded from this ticket. Helps
the agent avoid unsolicited refactoring.>

- <excluded item 1>
- <excluded item 2>

## Infrastructure

<Optional. Declarative list of runtime requirements that must already be
provisioned in the project for the implementation to be testable. The agent will
match each item against the project's docker-compose.yml, environment, or other
infra declarations and stop with a request to provision if any item is missing.>

- requires: postgres 15
- requires: redis 7
- requires: env var STRIPE_API_KEY

## Examples

<Optional. Concrete examples of inputs and expected outputs, useful for
disambiguating ambiguous criteria. Code blocks allowed.>

```python
>>> calc.subtract(5, 3)
2
```

## Notes

<Optional. Free-form notes for the agent: pointers to related code, prior art,
external references. Not a specification, just hints.>
```

### A.3 Field validation rules (used by FR-004)

The DoR check enforces the following rules and produces a structured failure report when any is violated:

| Rule | Condition for `READY` |
|---|---|
| Frontmatter parses as valid YAML | Required |
| Frontmatter contains `id` (non-empty slug, matches `^[a-z0-9][a-z0-9-]{2,63}$`) | Required |
| Frontmatter contains `title` (5 to 80 chars, non-empty) | Required |
| `## Description` section exists | Required |
| Description body is at least 50 non-whitespace characters | Required |
| `## Acceptance Criteria` section exists | Required |
| Acceptance Criteria contains at least one bullet matching `^- AC-\d+: .+$` | Required |
| Each acceptance criterion bullet has at least 10 non-whitespace chars after `AC-N:` | Required |
| If `## Infrastructure` section exists, each `requires:` line has a non-empty value | Required |
| File extension is `.md` | Required |
| File parses as Markdown without errors | Required |

### A.4 DoR comment format

When DoR fails, the agent appends a comment to the ticket file (or posts it to the linked issue) using this exact format:

```markdown
<!-- agent-code DoR report — DO NOT EDIT below this line -->

## DoR Report (agent-code)

**Status**: NOT_READY
**Generated at**: <ISO-8601 timestamp>
**Agent version**: <semver>

### Missing or insufficient fields

- **<field name>**: <one-line explanation of what is missing or insufficient>
- ...

### How to proceed

Edit this ticket to address the points above, then re-trigger `agent-code`.

<!-- end agent-code DoR report -->
```

### A.5 Example: a READY ticket

```markdown
---
id: add-subtract
title: Add a subtract function to calc
created: 2026-05-03
author: smorand
labels: [feature, calc]
---

# Add a subtract function to calc

## Description

The calc module currently exposes only `add`. We need a symmetric `subtract`
function for our internal arithmetic helpers, callable from any module that
imports `calc`.

## Acceptance Criteria

- AC-1: `calc.subtract(5, 3)` returns the integer `2`.
- AC-2: `calc.subtract(0, 0)` returns the integer `0`.
- AC-3: `calc.subtract` accepts two integer arguments and raises `TypeError`
  with a clear message on non-integer input.

## Out of Scope

- Floating-point support; integers only for this ticket.
```

### A.6 Example: a NOT_READY ticket and resulting DoR comment

A ticket that omits the Acceptance Criteria section entirely produces a DoR comment with the entry:

```
- **Acceptance Criteria**: section is missing. At least one bullet of the form `- AC-N: <criterion>` is required.
```

## Appendix B: Project Reference Template

This appendix defines the canonical Python project structure that ships with `agent-code` and is used by the bootstrap phase (FR-014). The template is the single source of truth for project conventions: every rule the agent applies during comprehension, planning, E2E writing, implementation, and review is documented in the template's `CLAUDE.md` and `.agent_docs/`. The agent itself carries no language-specific knowledge.

> **Implementation status**: this template is materialized in `vars/project-template/` of this repository. The descriptions below remain authoritative for structure and rules; the files in `vars/project-template/` are the canonical implementation. The bootstrap procedure (substitution + sync + `make check`) is documented in `vars/README.md` and has been validated end-to-end (89.58% coverage on a substituted project, all quality gates green).

### B.1 Top-level structure

```
<project-root>/
├── Makefile                      # Single interface for all dev commands
├── pyproject.toml                # uv + ruff + mypy + pytest config
├── uv.lock                       # Committed for reproducibility
├── .python-version               # Pinned Python version (3.11+)
├── .gitignore                    # Standard Python ignores + .agent_work/ exception
├── .pre-commit-config.yaml       # Optional: pre-commit hooks
├── README.md                     # Human-facing project doc
├── CLAUDE.md                     # Compact index for the agent (see B.4)
├── LICENSE                       # Required
├── Dockerfile                    # Multi-stage build template
├── docker-compose.yml            # Service orchestration template
├── .agent_docs/                  # Agent's deep documentation (see B.5)
│   ├── python.md
│   ├── makefile.md
│   ├── testing.md
│   ├── structure.md
│   ├── tooling.md
│   ├── ticket-template.md        # Mirror of Appendix A
│   └── pr-template.md            # Mirror of section 8.4
├── src/                          # Source directory, not a Python package
│   ├── py.typed                  # PEP 561 marker
│   ├── <project>.py              # Entry point
│   ├── config.py                 # pydantic-settings
│   └── logging_config.py
└── tests/
    ├── __init__.py
    ├── conftest.py
    ├── testdata/
    └── test_<entry>.py
```

### B.2 pyproject.toml requirements

The template's `pyproject.toml` declares:

- Build backend: `hatchling` (with `[tool.hatch.build.targets.wheel]`).
- `[project]` section: `name`, `version` (start at `0.1.0`), `description`, `requires-python = ">=3.11"`, `authors`, `license`.
- `[project.scripts]`: at least one CLI entry point.
- `[tool.uv]`: dev dependencies (pytest, pytest-cov, ruff, mypy, bandit).
- `[tool.ruff]`: enabled rule sets (E, F, I, B, UP, SIM, RUF), line length 100.
- `[tool.mypy]`: strict mode, no implicit `Any`, no untyped defs.
- `[tool.pytest.ini_options]`: `testpaths = ["tests"]`, `addopts = "-ra --strict-markers"`.
- `[tool.coverage.run]`: source = `["src"]`, branch = `true`.

### B.3 Makefile (single dev interface)

The template's Makefile MUST provide these targets verbatim. The agent invokes only these targets, never the underlying tools:

| Target | Purpose |
|---|---|
| `sync` | `uv sync` (install/refresh dependencies) |
| `run` | Execute the CLI entry point |
| `run-dev` | Run with debug logging and reload |
| `test` | `pytest` |
| `test-cov` | `pytest --cov=src --cov-report=term-missing` |
| `lint` | `ruff check src tests` |
| `lint-fix` | `ruff check --fix src tests` |
| `format` | `ruff format src tests` |
| `format-check` | `ruff format --check src tests` |
| `typecheck` | `mypy src` |
| `security` | `bandit -r src` |
| `check` | Run lint, format-check, typecheck, security, test-cov in order; exit non-zero on any failure |
| `build` | `uv build` |
| `install` | `uv tool install .` |

### B.4 CLAUDE.md (compact index, shipped with template)

The template's `CLAUDE.md` is the compact entry point. Its required sections, in order:

```markdown
# <Project Name>

> Compact AI agent index. Detailed docs live in `.agent_docs/`. Always read this
> file first; load `.agent_docs/*.md` only when relevant to the current task.

## What this project does

<One paragraph filled in by bootstrap from the ticket description.>

## Tech stack

- Python 3.11+
- uv (package manager)
- pytest (tests)
- ruff (lint and format)
- mypy (typing)
- bandit (security)

## Key commands

| Command | Purpose |
|---|---|
| `make sync` | Install dependencies |
| `make test` | Run tests |
| `make check` | Full quality gate (lint, format, type, security, tests with coverage) |
| `make run` | Run the CLI entry point |

**Always use `make`. Never invoke `uv`, `pytest`, `ruff`, or `mypy` directly.**

## Essential conventions

- `src/` is a source directory, not a Python package; no `src/__init__.py`.
- Imports use top-level module names, never the `src.` prefix.
- Tests mirror the source layout under `tests/`.
- Every new public function or class must have type annotations and a one-line docstring.
- Every commit must leave the branch with `make check` green.
- The agent never modifies files under `tests/test_*.py` once they are committed in the E2E phase. Fixtures under `tests/testdata/` and `tests/conftest.py` are exceptions.

## Documentation index

- `.agent_docs/python.md`: Python coding standards and idioms.
- `.agent_docs/makefile.md`: Detailed Makefile target reference.
- `.agent_docs/testing.md`: Testing conventions, fixtures, what counts as E2E.
- `.agent_docs/structure.md`: Project layout rules and file placement.
- `.agent_docs/tooling.md`: Available agent tools and when to use each.
- `.agent_docs/ticket-template.md`: Canonical ticket structure (mirror of spec Appendix A).
- `.agent_docs/pr-template.md`: Canonical PR description structure.

## Where to write what

- New module logic: under `src/`, mirroring an existing similar module.
- New tests: under `tests/`, mirroring the source layout.
- New project conventions discovered or decided during implementation: append to the relevant `.agent_docs/*.md` file and add an index entry here if a new file is created.
```

### B.5 .agent_docs/ contents (shipped with template)

Each file under `.agent_docs/` is a focused document under 200 lines. Required files and their content scope:

- **`.agent_docs/python.md`**: idiomatic Python (3.11+ features, async patterns, typing, error handling), forbidden patterns (mutable default args, broad `except`, unused imports), library preferences (httpx over requests, pydantic over dataclasses for IO, structlog or stdlib logging configured via `logging_config.py`).

- **`.agent_docs/makefile.md`**: per-target detail of what each Make target runs and what to do when it fails. Section "Reading `make check` output" with examples of common failure formats from ruff, mypy, bandit, pytest.

- **`.agent_docs/testing.md`**: how to write a pytest test (naming, fixtures, parametrize), what counts as E2E in this project, how to use `tests/testdata/`, how to use `conftest.py` fixtures, what mocking is allowed (`pytest-mock` for stdlib boundaries) and what is forbidden (mocking the unit under test). The rule "the agent never modifies `tests/test_*.py` once committed in the E2E phase" is restated here with rationale.

- **`.agent_docs/structure.md`**: directory and file placement rules (where to put new modules, when to split a file, when to create a subpackage), naming conventions (snake_case for modules, PascalCase for classes, UPPER_SNAKE for constants).

- **`.agent_docs/tooling.md`**: enumerates the read tools (`read_file`, `grep`, `find`, `ast_grep`, `lsp_*`), edit tools (`edit_file`, `write_file`, `apply_patch`), shell tool (`run_shell`), git tools, gh tools, MCP tools (`query_docs`, `search_web`). For each tool: purpose, when to prefer it, common pitfalls. Also documents the read-only constraint on `tests/test_*.py` during the implementation phase.

- **`.agent_docs/ticket-template.md`**: byte-for-byte mirror of Appendix A, sections A.1 through A.6.

- **`.agent_docs/pr-template.md`**: byte-for-byte mirror of section 8.4 (Pull Request description format), plus rules: every section must be populated, no empty sections, no placeholder text. References to E2E tests use the `path::function` pytest notation.

### B.7 .gitignore policy for .agent_work

The template's `.gitignore` does NOT exclude `.agent_work/`. Per the architectural decision in section 14, `.agent_work/` is tracked in git on the feature branch and survives the squash-merge as a single artifact in the PR's diff (then collapses cleanly after merge to main).

### B.8 Template versioning

The template ships with a `.template_version` file at its root containing a semver string. Future versions of `agent-code` may detect projects with stale templates and propose an upgrade, but auto-upgrading is out of MVP scope.
